# OpenSportsData 开源赛事数据聚合索引

OpenSportsData 是一个面向体育数据分析师、赛事运营团队及开源情报研究社区的技术资源聚合项目。本项目不提供数据存储或原始数据抓取服务，而是围绕公开赛事信息源构建结构化导航体系，提供稳定的外链索引、域名状态参照与数据源分类框架，帮助技术团队快速定位可用数据入口，降低信息检索与维护成本。项目定位为轻量级、可自托管的导航中间件，适用于数据管道前置层、爬虫调度参照系或自动化监控系统的源头配置。

## 功能概览

- **赛事数据源分类索引** 按赛事类型、地域、数据粒度对公开数据入口进行层级划分，支持按需筛选。
- **域名可用性参照面板** 提供数据源域名的当前解析与访问状态参考，辅助诊断链路健康度。
- **多级导航结构生成** 基于配置文件自动生成多级目录页，支持静态部署，无需后端服务。
- **数据源变更追踪提示** 通过对比历史配置与当前状态，标注新增、失效或跳转的域名，便于维护。
- **轻量级监控钩子** 支持集成外部健康检查工具，对关键域名进行定时探测并输出结构化日志。
- **配置即代码管理** 所有数据源列表、分类标签、备注信息均通过 YAML 或 JSON 文件管理，便于版本控制。
- **纯静态输出设计** 构建过程生成纯 HTML 与 Markdown 页面，可托管于任意 Web 服务器或对象存储。

## 应用场景

- **数据管道前置配置** 数据工程师在构建 ETL 流程时，可通过本项目提供的域名清单快速配置爬虫入口或 API 网关，避免手动整理散落的数据链接。
- **赛事运营监控面板** 运营团队可将本项目部署为内部监控面板的导航层，实时参照各数据源的访问状态，辅助判断数据延迟或源站故障。
- **开源情报研究基础** 研究人员可使用本项目作为赛事数据检索的起点，快速定位不同赛程类型的数据发布入口，减少重复搜索时间。
- **自动化测试用例维护** 测试团队可利用本项目提供的稳定域名列表，编写连通性测试用例，在 CI/CD 流程中自动验证外部依赖的可用性。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，需要 Git 与 Node.js 18+ 或 Python 3.10+ 运行环境（根据实际发行版本选择）。

```bash
# 克隆项目仓库
git clone https://github.com/opensportsdata/opensportsdata-index.git
cd opensportsdata-index

# 安装依赖（以 Node.js 版本为例）
npm install

# 构建静态导航页与索引文件
npm run build

# 启动本地预览服务（默认端口 8080）
npm run serve
```

访问本地服务后，将自动生成包含所有分类数据源的导航首页。若使用 Python 版本，请参考 `docs/python-setup.md` 中的对应命令。

## 项目结构

项目采用模块化分层设计，构建产物与源代码分离，便于维护与二次开发。以下为关键目录与文件说明：

```
opensportsdata-index/
├── config/                        # 配置目录，存放所有数据源定义
│   ├── sources.yaml               # 主数据源清单，含域名、分类、备注
│   ├── categories.yaml            # 分类层级与显示顺序定义
│   └── monitoring.yaml            # 健康检查阈值与超时配置
├── src/                           # 源代码目录
│   ├── builder/                   # 构建核心逻辑
│   │   ├── index.js               # 入口脚本，协调构建流程
│   │   ├── page-generator.js      # 生成 HTML 与 Markdown 页面
│   │   └── link-validator.js      # 域名格式校验与去重工具
│   ├── templates/                 # 页面模板文件（EJS / Handlebars）
│   │   ├── index.template.html    # 首页导航模板
│   │   └── detail.template.html   # 分类详情页模板
│   └── utils/                     # 通用工具函数
│       ├── logger.js              # 日志输出格式化
│       └── fs-helper.js           # 文件读写封装
├── public/                        # 静态资源目录（复制到构建输出）
│   ├── css/                       # 样式文件
│   │   └── style.css              # 主样式表（响应式设计）
│   └── js/                        # 前端交互脚本
│       └── filter.js              # 客户端分类筛选逻辑
├── dist/                          # 构建输出目录（自动生成，不纳入版本库）
│   ├── index.html                 # 生成的首页导航页
│   └── categories/                # 各分类详情页
├── tests/                         # 单元测试与集成测试
│   ├── builder.test.js            # 构建逻辑测试
```

## 贡献指南

我们欢迎各类贡献，包括但不限于新增数据源、改进文档、修复构建问题或优化页面样式。请遵循以下步骤：

1. 在 GitHub 上 Fork 本仓库，并克隆到本地开发环境。建议使用独立分支进行开发，分支命名格式为 `feature/功能简述` 或 `fix/问题描述`。
2. 修改 `config/sources.yaml` 文件时，请确保新增的域名符合格式规范（不含协议头、不含路径），并填写完整的分类标签与备注信息。提交前运行 `npm run validate` 校验配置合法性。
3. 若涉及页面模板或样式改动，请使用 `npm run dev` 启动开发模式，实时预览效果，并在不同屏幕尺寸下验证响应式表现。确认无误后运行 `npm run test` 确保现有测试用例通过。
4. 提交 Pull Request 时，请填写清晰的变更摘要，说明修改动机与影响范围。若新增数据源，请附上公开可查的来源说明，以便维护者审核。
5. 重大变更（如目录结构调整、构建流程重写）请先通过 Issue 与维护者沟通，避免重复劳动或架构冲突。

## 常见问题

**问：项目是否提供原始赛事数据或 API 接口？**

本项目为纯导航与索引服务，不存储、缓存或代理任何原始数据内容。所有列出的域名均为公开可访问的信息发布入口，用户需自行遵守各源站的使用条款与 robots 协议。项目本身不提供数据查询 API，仅输出静态导航页面与配置清单。

**问：部分域名无法访问或解析失败，应如何处理？**

域名可用性受外部因素影响，包括源站维护、网络策略调整或 DNS 变更。用户可先通过本地 `dig` 或 `ping` 工具确认网络环境。若确认域名已永久失效或更换，欢迎通过 Issue 或 Pull Request 提交更新，维护团队将定期合并社区反馈并发布新版本配置。项目自身的监控钩子仅做参考，不构成服务可用性承诺。

**问：能否将本项目部署到内网环境，不连接外网？**

可以。项目构建过程完全在本地执行，无需外网依赖（npm 安装包除外）。构建完成后生成的 `dist/` 目录为纯静态文件，可复制到任意内网 Web 服务器。但需注意，导航页中列出的数据源域名本身需要内网能够解析访问才可正常使用。若需完全离线使用，建议将数据源替换为内网镜像地址。
