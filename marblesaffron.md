# ResourceLink Navigator

ResourceLink Navigator 是一个面向开发者、技术研究人员与信息架构师的高质量外链资源整合平台。本项目不直接托管或存储任何第三方内容，而是通过结构化索引、可追溯的链接分类与版本化外链管理机制，为技术团队提供可靠、可维护、可审计的外部资源引用基础设施。

本项目核心目标用户包括：文档工程师、技术社区运营者、开源项目维护者、自动化测试脚本编写人员，以及任何需要在大规模信息环境中持续维护外链有效性的技术角色。ResourceLink Navigator 通过标准化的资源描述格式、自动化的链接健康检查脚本与清晰的目录分层，有效降低外链失效、域名迁移与内容变体带来的信息腐化风险，显著提升技术文档与知识库的长期可维护性。

## 功能概览

- **结构化外链索引引擎**：支持按资源类型、业务领域、域名后缀与内容语种对原始链接进行多维度自动归类，生成可读性良好的分类目录。

- **链接健康状态监测**：内置轻量级 HTTP 状态检查脚本，可定期探测每个收录链接的可达性、重定向链与响应时间，输出结构化健康报告。

- **版本化资源快照**：每次收录或更新操作均生成带时间戳的清单快照，支持回溯任意历史版本的资源集合，便于审计与差异对比。

- **自定义元数据标注**：允许维护者为每个链接添加标签、备注、维护等级与预期更新频率，形成超越纯 URL 列表的增强型资源元数据库。

- **多格式导出接口**：支持将资源列表导出为 JSON、YAML、CSV 及纯文本格式，便于下游 CI/CD 管道、静态站点生成器或自动化测试套件直接消费。

- **链接变动通知机制**：当检测到链接状态变更（如 200 变为 404、301 永久重定向、域名证书过期等）时，可通过日志文件与控制台输出发出明确告警。

- **依赖关系透视功能**：若链接之间存在引用链或逻辑依赖（如 A 站指向 B 站的 API 文档），本系统可记录并可视化这些关系，辅助影响面分析。

## 应用场景

- **技术文档库的外链资产管理**：大型开源项目的文档站点往往包含数百个外部引用，ResourceLink Navigator 可帮助维护团队集中管理这些链接，并在版本发布前自动检查所有外链的有效性，避免用户点击失效链接。

- **爬虫与自动化测试的环境配置中心**：自动化测试脚本经常需要依赖外部测试环境或 Mock 数据源，本系统可作为统一的资源注册表，测试执行前拉取最新链接清单，确保测试环境配置与线上实际资源同步。

- **技术资讯聚合站的后台链接整理**：面向开发者的每日资讯或周报类站点，需要从多个来源聚合内容，本系统提供清晰的外链分类与状态追踪能力，大幅降低人工整理与核对成本。

- **内部知识库的引用规范化改造**：企业或组织内部的技术 wiki 常存在大量格式不统一、来源不明的外链，使用本系统可逐步迁移并规范所有外部引用，形成统一的资源入口层。

- **开源社区贡献者入门指引**：新加入社区的贡献者往往不清楚该参考哪些权威外部资料，本系统可作为官方推荐的资源索引，帮助新人快速定位高质量学习材料与工具站点。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，需提前安装 Git 与 Node.js 18.x 或以上版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcelink-navigator/core.git resourcelink-navigator
cd resourcelink-navigator

# 安装项目依赖（使用 npm）
npm install

# 执行首次资源索引构建，生成初始资源清单
npm run build:index
```

执行完成后，可在 `output/` 目录下找到生成的资源清单文件 `resource-index.json` 与 `resource-index.yaml`。

## 贡献指南

我们欢迎并鼓励社区贡献，包括但不限于新增资源链接、优化分类规则、改进健康检查算法、完善文档以及报告问题。请遵循以下步骤参与贡献：

1. **创建议题讨论**：在提交拉取请求之前，请先在 Issues 区域创建一个议题，简要描述你希望解决的问题或新增的功能。此举可避免重复工作，并确保改动方向与项目维护者达成一致。

2. **派生仓库并创建特性分支**：从主仓库派生一份副本至你的个人账户，然后在本地切换到新分支，分支命名建议采用 `feature/xxx` 或 `fix/xxx` 格式，例如 `feature/add-sports-resources`。

3. **实施改动并完善测试**：在分支上完成代码或文档修改。若涉及代码逻辑变更，请补充对应的单元测试或集成测试，确保现有测试套件全部通过。若为资源列表增删，请同步更新 `data/raw/` 下的批次文件与 `output/` 目录的预期快照。

4. **编写清晰提交信息**：提交时使用简明扼要的提交信息，推荐采用约定式提交格式，例如 `feat: add new resource batch 002` 或 `fix: correct redirect handling in http-client`。

5. **发起拉取请求并参与评审**：将分支推送至你的派生仓库，然后向主仓库的 `main` 分支发起拉取请求。在请求描述中引用相关议题编号，并逐项说明改动内容与测试结果。项目维护者将在 3 个工作日内进行评审，必要时会提出修改意见。

## 常见问题

**问：如果收录的某个链接永久关闭或变更了域名，我应该如何处理？**

答：当链接永久不可访问时，请先在 Issues 中提交链接失效报告，附上探测时间、HTTP 状态码及任何可见的重定向目标。项目维护者将核实后从活跃索引中移除该链接，并将其移动至 `data/archived/` 目录下的历史清单中，同时更新快照版本号。若链接仅发生域名迁移但内容主体未变，则建议提交拉取请求，将旧链接替换为新域名，并在提交信息中注明迁移原因。

**问：健康检查脚本是否会对目标站点造成过大的请求压力？**

答：不会。健康检查默认采用串行请求模式，且每个请求之间强制间隔 500 毫秒，超时时间设置为 5 秒。同时，检查频率默认为每 24 小时执行一次，仅对 HEAD 请求优先，仅当 HEAD 失败时才回退至 GET 请求。对于批量新增的链接，首次检查将随机打乱顺序执行，避免对单一站点短时间内发起连续请求。如需调整上述参数，可修改 `config/default.yaml` 中的相关配置项。

**问：能否在无法访问公网的内网环境中使用本系统？**

答：可以。本系统的核心索引构建与元数据管理功能完全离线可用，仅健康检查模块需要网络访问。若内网环境完全隔离，你可以禁用健康检查功能（通过 `npm run build:index -- --no-check`），并将本系统纯粹作为外链的本地注册表与分类管理工具使用。此外，你也可以将健康检查的请求代理配置为内网代理地址，具体方法参见 `docs/ops/health-check.md` 中的代理配置章节。
