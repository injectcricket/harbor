# DSZQ Resource Hub

DSZQ Resource Hub 是一个面向中文技术社区的开源外链与资源导航系统，专为需要快速检索、分类管理和批量访问特定领域在线资源的开发者、运维人员及技术决策者设计。该项目不提供具体内容托管，而是以高可读性的结构化索引方式，将分散的优质外部链接按主题、用途和访问频率进行组织，从而降低信息查找成本，提升工作流中的资源获取效率。

本项目适用于个人技术知识库的补充、团队内部文档系统的外链整合层，以及小型组织在内部网络环境中搭建轻量级资源聚合页。通过清晰的目录树、固定的章节模板和纯静态化的部署方式，DSZQ Resource Hub 能够在不依赖数据库的情况下，实现接近零维护成本的资源导航服务。

---

## 功能概览

- **多级分类索引**：支持按技术领域、服务类型、区域归属等维度对链接进行树状归类，便于用户按业务逻辑逐层定位。
- **原始链接透传**：所有外链均以纯文本代码块形式原样呈现，杜绝自动补全协议或域名改写，保证目标地址的精确可访问性。
- **批量导入与校验**：提供脚本工具，支持从 CSV 或 JSON 文件批量导入链接列表，并自动校验域名可达性与协议一致性。
- **静态站点生成**：基于模板引擎一次性渲染全站 HTML，无需运行时动态查询，可直接部署于 Nginx、Apache 或对象存储服务。
- **自定义元数据扩展**：每个链接条目允许附加描述标签、更新时间和状态标记（如有效、失效、维护中），便于后期人工复核。
- **访问频率统计接口**：集成轻量级日志分析模块，可输出各链接被点击的次数排序，辅助优化首页推荐位。
- **暗色与高对比度主题**：内置两套 CSS 变量方案，满足不同光照环境下的阅读需求，同时兼顾视觉障碍用户的可用性。

## 应用场景

- **企业内部技术文档门户**：将研发团队常用的 API 文档、监控面板、日志系统、代码仓库等外部链接统一收录，作为新员工入职指引的一部分，减少反复询问内部地址的频率。
- **开源项目外部依赖索引**：当项目需要引用大量第三方库、数据源、参考实现或在线测试工具时，使用本项目的索引结构维护这些依赖的原始访问地址，避免版本升级后链接散落丢失。
- **个人知识管理辅助层**：对于使用 Markdown 或 Org-mode 维护知识库的开发者，可将本项目生成的链接目录作为独立模块嵌入，实现笔记内容与外部参考源的清晰分离。
- **网络隔离环境下的白名单准备**：在需要配置防火墙或代理规则的内网环境中，先通过本项目整理出完整的目标域名列表，再统一提交给网络管理员进行放行或路由配置。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，要求已安装 Git 和 Node.js 18 以上版本。

```bash
# 克隆仓库
git clone https://github.com/dszq-hub/resource-hub.git
cd resource-hub

# 安装依赖
npm install

# 生成静态站点（默认输出至 ./dist 目录）
npm run build

# 启动本地预览服务（默认端口 8080）
npm run serve
```

执行完毕后，打开浏览器访问 `http://localhost:8080` 即可查看生成的资源导航页面。若需更新链接列表，请编辑 `./data/links.json` 文件后重新运行 `npm run build`。

## 贡献指南

1. **分支规范**：请基于 `develop` 分支创建功能分支，命名格式为 `feature/功能简述` 或 `fix/问题编号`。提交前确保当前分支与 `develop` 保持同步。
2. **链接增改流程**：若需添加或修改外部链接，请编辑 `data/links.json` 文件，并严格按照 `docs/data-schema.md` 中定义的字段类型填写。新增条目需包含 `title`、`url`、`category` 和 `status` 四个必填字段。
3. **提交信息格式**：遵循 Conventional Commits 规范，提交消息首行须包含类型（如 `feat:`、`fix:`、`docs:`）和简要描述，正文可补充更详细的改动说明。
4. **测试要求**：所有对核心逻辑（`src/core/` 目录下文件）的修改，需在 `tests/` 目录下补充或更新对应的单元测试用例，并确保 `npm test` 全部通过后方可发起 Pull Request。
5. **文档同步**：若改动涉及用户可见的行为（如新增配置项、修改命令行参数），必须同步更新 `docs/` 目录下的相关手册，并检查 README 中是否有需要对应的调整。

## 常见问题

**问：构建时提示 "ValidationError: Invalid URL format"，但我的链接明明可以正常访问，如何解决？**

答：本项目的校验器默认要求链接必须包含明确的协议头（http:// 或 https://），且不允许出现末尾斜杠或非 ASCII 字符。请检查你的条目中 `url` 字段是否遗漏协议前缀，或包含不可见空格。若确需使用裸域名，可修改 `src/core/validator.js` 中的正则规则，但需注意这会影响所有条目的校验逻辑。

**问：我添加了新链接并重新构建，但预览页面没有显示任何变化，是什么原因？**

答：首先确认 `npm run build` 命令是否成功执行且无报错。其次检查 `dist/` 目录下生成的 HTML 文件的修改时间是否更新。若仍无变化，请清除浏览器缓存（Ctrl+Shift+R 强制刷新），或检查 `data/links.json` 中的 `category` 字段是否与模板中使用的分类过滤名称一致。

**问：能否将项目部署到 GitHub Pages 或类似的静态托管服务？**

答：完全可以。执行 `npm run build` 后，将 `dist/` 目录下的所有文件上传至目标托管平台即可。若使用 GitHub Pages，请将构建命令配置为 CI 工作流的一部分，或者手动将 `dist/` 内容推送到 `gh-pages` 分支。
