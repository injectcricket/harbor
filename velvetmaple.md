# HyperLink Index

HyperLink Index 是一个面向技术调研、数据聚合与外部资源管理的轻量级链接导航系统。项目定位为开发者与研究人员提供结构化、可维护的外部 URL 索引方案，解决多源异构链接在文档、项目或团队协作中难以统一管理、难以版本追踪、难以快速检索的痛点。目标用户包括开源项目维护者、技术文档工程师、数据采集与清洗人员，以及需要长期维护大量外链资源的站点运营者。

本系统不提供内容代理、不修改外部资源、不存储第三方页面数据，仅作为链接元信息的组织与展示层。通过标准化目录结构、Markdown 原生渲染与静态站点生成机制，HyperLink Index 可在任何支持 HTTP 文件服务的环境中运行，无需数据库或动态后端。项目本身以开源资源汇总站为原型，同时可作为通用外链管理模板，适用于技术导航、学术参考、工具清单、数据源登记等多种场景。

## 功能概览

- **链接分级分类存储**：支持按主题、批次、来源机构等多维度建立目录层级，每个链接条目可附加标签、说明文本与状态标记。
- **原始 URL 原样呈现**：系统严格保留用户提交的 URL 字符串，不自动补全协议、不添加或移除 www 前缀、不修改大小写、不追加尾部斜杠，确保链接精确指向预期资源。
- **Markdown 清单驱动**：所有外链数据以 Markdown 列表或表格形式记录于项目仓库中，便于 Git 版本追踪、差异对比与协作审阅。
- **自动目录树生成**：基于文件系统结构自动生成 ASCII 目录树注释，帮助维护者快速了解资源分布与各目录职责。
- **多场景导航模板**：内置技术文档导航、赛事数据索引、区域服务名录等典型页面模板，可快速适配不同业务领域。
- **依赖极简部署**：仅依赖标准 Python 3 环境与常见文本处理工具，无需额外数据库或容器编排系统。
- **批量校验与报告**：提供链接可达性检查脚本，可定期输出失效链接报告，辅助维护者清理或更新过期资源。

## 应用场景

- **赛事数据源整理**：体育数据分析团队可将多个比分、赛程、排名类外部站点统一收录，按联赛或赛季建立索引目录，供内部数据抓取流程引用，避免硬编码散落在各采集脚本中。
- **技术文档外链管理**：开源项目文档中常需引用大量第三方规范、SDK 下载页、API 参考或社区讨论帖。使用 HyperLink Index 可将这些外链集中维护，文档正文仅引用索引条目 ID，后期更换链接时只需修改一处。
- **地区服务节点登记**：跨区域部署的系统运维团队可将不同地域的监控面板、日志查看器、配置中心地址按区域归类，形成可公开或内网可见的服务导航页，降低新成员上手时的信息查找成本。
- **学术参考文献链接库**：研究机构可将论文中引用的数据集主页、工具仓库、预印本服务器、机构公告页等整理为结构化清单，作为论文附件或项目仓库的独立章节发布，提升研究成果的可复现性。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，假设已安装 Git 与 Python 3.8 及以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/example/hyperlink-index.git
cd hyperlink-index

# 安装依赖（核心依赖仅标准库，额外校验工具需安装 requests）
pip install requests

# 运行本地开发服务器（默认端口 8000）
python -m http.server 8000

# 或使用项目内置的静态生成脚本，输出至 dist 目录
python build.py --input ./data --output ./dist
```

启动后，在浏览器中访问 `http://localhost:8000` 即可查看默认导航页面。如需自定义资源列表，请编辑 `./data/links.md` 文件并按格式添加条目，随后重新运行构建脚本。

## 贡献指南

1. 复刻项目仓库至个人账户，在本地新建功能分支（如 `feature/add-batch-125`），确保分支命名体现变更意图。
2. 在 `data/batches/` 目录下新增对应批次号的 Markdown 文件，按现有格式逐行列出 URL，每条占一行，URL 前后无需添加额外符号。若涉及分类调整，同步修改 `data/categories/` 下的视图文件或 `meta.json` 中的分类映射。
3. 运行本地构建脚本 `python build.py --strict` 验证输出是否正常，同时执行 `scripts/check_links.py --batch 125` 对新加入的链接进行基础可达性检测（超时阈值 5 秒，失败重试 1 次）。
4. 提交变更时使用清晰的信息体，例如 `docs: add batch 125 with 12 resource URLs` 或 `fix: correct domain spelling in batch 123`。提交前确保所有测试脚本通过，且未引入语法错误或路径错误。
5. 发起拉取请求至主仓库的 `main` 分支，在请求描述中注明新增批次编号、链接总数以及任何特殊注意事项（如部分站点需特定 User-Agent 或存在访问频率限制）。

## 常见问题

**问：为什么系统不自动将裸域名转换为 `https://` 或 `http://` 格式？为什么不自动添加或移除 `www` 前缀？**

答：某些外部资源对协议或子域名有严格区分（例如 `http://` 与 `https://` 可能指向不同服务，`www.example.com` 与 `example.com` 可能对应不同应用或地域节点）。自动补全或标准化会引入不可预见的重定向或访问失败风险。HyperLink Index 的核心理念是“原样存储，原样展示”，由用户保证所提交 URL 的有效性与精确性。

**问：如何批量更新已有链接的说明文字或分类标签？**

答：推荐使用 `scripts/import_batch.py` 脚本的更新模式（`--update` 参数），该模式会读取 `data/meta.json` 中记录的原始条目，并与新传入的 CSV 文件进行 ID 匹配，仅覆盖说明字段与分类字段，保留 URL 原始字符串不变。若需要彻底替换某批次内容，可直接编辑对应的批次 Markdown 文件，然后重新运行完整构建流程。

**问：链接可达性检查脚本产生大量误报（如站点启用了反爬或临时维护），如何处理？**

答：检查脚本支持通过 `--exclude` 参数忽略特定域名或 URL 模式，同时可在 `scripts/check_links.py` 顶部配置 `USER_AGENT` 和 `REQUEST_HEADERS` 常量以模拟常见浏览器标识。对于频繁变更或已知不稳定的站点，建议在 `meta.json` 中为该条目设置 `"check_enabled": false` 标记，跳过自动化检查，转为人工定期复核。
