# Hyperlink Nexus

Hyperlink Nexus 是一个面向技术开发者与开源项目维护者的外链资源聚合与规范化管理工具。该项目并非传统的内容管理系统，而是一个轻量级、可自托管的链接治理中间件，用于对分散在多源渠道中的外部 URL 执行去重、有效性校验、分类标注、版本追踪与批量导出。目标用户包括技术文档编写者、社区运营人员、自动化巡检脚本开发者，以及需要长期维护大量外链引用关系的项目维护人。

Hyperlink Nexus 解决的核心问题是：当项目文档、技术博客、教学提纲或运维手册中散落成百上千条外部链接时，如何以工程化手段保证这些链接的可用性、一致性与可追溯性。项目提供纯静态化的链接清单渲染能力，并支持通过结构化元数据对链接按业务领域、访问频率、来源批次、状态码历史等维度进行二次编排，最终输出为机器可读或人类可查阅的 Markdown 汇总文档。

## 功能概览

- **链接批量入库与去重**：支持从文本文件、CSV 或直接粘贴的原始 URL 列表中自动解析并剔除重复项，保留首次出现的时间戳与来源标记。

- **协议与域名规范化校验**：自动识别裸域名、带协议头、带 www 前缀等不同书写风格，按照用户预设策略统一存储形态，但对外展示时严格遵循用户原始输入格式。

- **自定义分类标签体系**：允许为每条链接附加一个或多个分类标签（如 "官方文档"、"社区论坛"、"API 参考"、"数据集"），并支持按标签组合筛选生成子清单。

- **状态巡检与可用性报告**：内置基于 HTTP 头部探测的轻量级链接存活检查功能，可生成失效链接列表并记录重定向链变化，辅助维护人及时发现断裂引用。

- **多格式清单导出**：除 Markdown 表格与列表外，支持输出为 JSON 结构化数据、YAML 配置格式，以及适用于静态站点生成器的 frontmatter 摘要文件。

- **批次管理与变更日志**：以批次编号（如第 86/150 批）为组织单位记录每批链接的入库时间、来源说明与操作人，便于追溯批量改动历史。

- **只读只写分离的权限模式**：提供简单的环境变量控制，支持在生产环境关闭写入接口，仅开放只读查询与导出能力，降低误操作风险。

## 应用场景

- **开源项目 README 外链维护**：项目维护者可将所有引用的教程、工具站、镜像源、参考文献等 URL 统一托管于 Hyperlink Nexus，每次发布新版本前自动校验一遍有效性，避免用户点击到过期域名。

- **技术文档门户的资源聚合页生成**：文档团队可将不同章节中散落的参考链接集中导入系统，按主题分类后生成一份独立的“资源列表”页面，既减轻正文冗余，又提升读者查找效率。

- **社区周报或月刊的链接素材管理**：运营人员将本周收集到的活动报名页、视频回放、幻灯片下载地址、讨论区热帖等一次性录入，系统自动排序去重并输出整洁的汇总清单，直接粘贴至邮件或论坛帖子中。

- **依赖项与第三方服务的变更影响分析**：当某个外部服务更换域名或弃用旧 API 时，管理员可快速检索出所有引用该域名的链接列表，评估影响范围并逐一更新。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，需提前安装 Git、Node.js（v18 以上）与 npm。

```bash
# 克隆项目仓库
git clone https://github.com/hyperlink-nexus/hn-core.git
cd hn-core

# 安装依赖项
npm install

# 使用示例数据启动开发服务（默认监听 3000 端口）
npm run start:dev
```

启动成功后，访问控制台输出的本地地址即可看到示例链接看板。如需导入用户自定义链接，请将原始 URL 列表按每行一条的格式放入 `data/raw/batch_86_150.txt` 文件，随后执行：

```bash
npm run ingest -- --batch=86/150
```

生成的汇总文档将输出至 `output/` 目录。

## 贡献指南

1. 在 GitHub 上 fork 本仓库至个人账号，并克隆到本地开发环境。建议在新建分支上工作，分支命名遵循 `feature/功能简述` 或 `fix/问题描述` 的格式。

2. 阅读 `docs/developer/` 目录下的开发环境搭建文档，运行 `npm install` 和 `npm run test:unit` 确保现有测试全部通过，以确认本地环境配置正确。

3. 如新增功能或修改核心逻辑，请同步补充对应的单元测试用例，测试覆盖率不应低于现有基线（80%）。修改后执行 `npm run lint` 与 `npm run format` 统一代码风格。

4. 提交代码时使用语义化提交信息（如 `feat: add batch import from CSV` 或 `fix: correct domain normalization for bare hostnames`），并推送到 fork 仓库。

5. 通过 GitHub Pull Request 向主仓库的 `main` 分支发起合并请求，在 PR 描述中清晰说明改动目的、影响范围以及是否包含破坏性变更。核心维护者会在 3 个工作日内评审。

## 常见问题

**问：导入链接时，系统是否会自动补全裸域名的协议头？**

答：不会。系统内部存储时会保留用户输入的原始字符串形态，同时在输出环节严格遵循“原样输出”原则。校验阶段仅检查域名是否可解析，但不会修改用户提供的书写格式。用户如需统一样式，可在导入前通过预处理脚本自行处理。

**问：存活探测功能是否会频繁访问外部链接，导致被目标网站封禁？**

答：探测模块默认启用延迟间隔（500 毫秒）和重试上限（2 次），并且仅发送 HEAD 请求而非 GET 请求，以减少带宽消耗。同时支持通过环境变量 `PROBE_RATE_LIMIT` 调整每秒最大请求数。对于生产环境的大规模链接库，建议配合 `--dry-run` 模式先输出待探测列表，评估后再执行实际探测。

**问：如何迁移或备份已录入的链接数据？**

答：所有链接元数据（包括原始文本、分类标签、入库时间戳、探测结果）均以 JSON Lines 格式存储于 `data/cache/` 目录下的本地文件中，无外部数据库依赖。备份时直接复制整个 `data/` 目录即可。恢复时将其放回原路径，重启服务后自动加载。
