# HyperLink Compass

HyperLink Compass 是一个面向技术调研人员、数据分析师与开源生态观察者的高质量外链聚合导航系统。本项目不生产原始内容，而是通过人工筛选与自动化校验相结合的方式，将分散于各垂直领域的权威数据源、实时信息面板与结构化统计资源整合至统一入口，解决信息检索碎片化、来源可信度难以评估以及关键数据延迟获取的三大痛点。系统内置链接可用性监控与变更告警机制，确保每条聚合资源均保持有效可访问状态，适用于需要高频依赖外部公开数据源的技术团队与个人研究者。

## 功能概览

- **垂直领域资源聚合**：按竞赛类型、统计主体与数据时效性对原始外链进行一级分类，支持快速定位特定赛事或排行榜资源。
- **链接健康度探针**：每日定时对收录的每个 URL 执行 TLS 证书有效性、响应状态码与页面标题匹配三项检查，异常时通过日志输出告警。
- **元信息自动补全**：基于 HTTP 响应头与 HTML meta 标签，自动提取每个资源的站点名称、最后更新时间和内容语言，减少人工标注成本。
- **只读镜像缓存**：对关键统计页面生成静态文本快照（不含图片与脚本），用于源站临时不可用时的备查降级方案。
- **变更差异对比**：对支持 If-Modified-Since 或 ETag 的资源端点，记录每次抓取的内容哈希，生成变更摘要供人工审阅。
- **标签化检索体系**：允许用户按机构后缀（.org.cn / .com / .net）、数据类别（赛果 / 积分榜 / 实时比分）以及更新频率（每日 / 每周）进行组合筛选。
- **外部引用跟踪**：记录每个资源在项目内被访问的次数与最近访问时间，辅助识别高价值数据源以优化缓存策略。

## 应用场景

- **体育数据产品原型开发**：产品经理或前端工程师在构建体育资讯类 Demo 时，可使用本系统提供的现成赛果与积分榜 URL 作为数据填充源，避免因缺乏真实接口而阻塞界面设计与交互验证进度。
- **多源数据交叉验证**：数据分析师在对同一赛事的不同统计口径进行一致性核验时，可通过本系统快速访问多个独立来源的比分与排名页面，手动或脚本化对比数值差异，识别异常记录。
- **外链资源迁移审计**：当组织内部的知识库或门户网站需要批量更新外部引用链接时，运维人员可利用本系统的资源清单与健康度日志，优先处理即将失效或已变更的 URL，降低用户访问失败率。
- **学术研究样本采集**：社会科学或体育管理领域的研究生在开展基于公开赛事数据的纵向分析时，可将本系统作为样本来源导航，系统化记录不同时间段的数据页面 URL，确保研究方法可复现。

## 快速开始

以下步骤适用于 Linux / macOS 环境，需提前安装 Git 与 Node.js（v18 及以上）。

```bash
# 克隆仓库至本地
git clone https://github.com/your-org/hyperlink-compass.git
cd hyperlink-compass

# 安装项目依赖（使用 npm）
npm install

# 启动本地开发服务（默认监听 3000 端口）
npm run dev
```

执行上述命令后，访问 `http://localhost:3000` 即可查看资源导航面板。生产环境部署请使用 `npm run build` 配合 `npm run start`。

## 贡献指南

本系统欢迎社区提交资源增补、探针逻辑优化以及文档改进。请遵循以下步骤以确保合并效率：

1.  **查阅现有议题**：访问 GitHub Issues 页面，搜索是否已有相同资源新增请求或探针缺陷报告。若无重复，请新建一个议题描述您的修改动机与预期效果。
2.  **派生仓库并创建分支**：从主仓库的 `main` 分支派生（Fork）至个人账户，然后在本地的派生副本中创建一个新分支，分支命名建议采用 `feature/资源类别-名称` 或 `fix/探针-问题简述`。
3.  **实施修改并自测**：若涉及新增资源 URL，请同步更新 `config/default.yaml` 中的 `resources.list` 字段，并运行 `npm run test` 确保所有既有探针测试与解析测试通过。若调整前端界面，请附带截图说明。
4.  **提交变更并推送**：提交信息（Commit Message）请使用约定式提交规范（如 `feat: 新增哈萨克超实时比分资源` 或 `fix: 修复内容哈希计算时未忽略换行符的问题`），随后推送到您的派生仓库。
5.  **发起拉取请求**：在 GitHub 上向主仓库的 `main` 分支发起拉取请求（Pull Request），并在描述中关联对应的议题编号。维护者将在三个工作日内进行审阅，期间可能会请求补充测试用例或调整实现细节。

## 常见问题

**Q：为什么某些 URL 在列表中显示为裸域名（如 `<code>fenchaobisaijieguo.org.cn</code>`），而不是完整的 `https://` 链接？访问时是否需要手动补全协议？**

A：本系统严格保留用户提供的原始 URL 格式，旨在忠实反映资源来源的原始记录。实际访问时，现代浏览器通常会自动尝试 HTTPS，但若遇到证书错误或连接超时，建议您手动尝试 `http://` 前缀。本项目的健康度探针会同时记录 HTTP 与 HTTPS 的连通性结果，您可在仪表盘中查看详细报告。

**Q：探针检测到某个链接失效后，系统是否会主动通知我？**

A：当前版本的通知机制仅输出至日志文件（`logs/probe-report-日期.log`）以及 Web 面板的“异常资源”区域，暂未集成邮件或即时通讯推送。您可以通过编写外部脚本定时解析日志文件来实现自定义告警。我们计划在 v2.0 版本中增加 Webhook 对接能力。

**Q：我可以将这些资源列表用于商业产品吗？**

A：本系统仅提供导航与元信息聚合功能，所有原始 URL 指向的页面内容版权归其各自所有者。使用这些资源时，请遵守目标站点的服务条款。本项目代码（即本仓库中除 `config/default.yaml` 中的资源列表外的所有部分）采用 MIT 许可证，您可以自由使用、修改和分发，但需保留原始版权声明。
